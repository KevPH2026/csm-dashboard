## 一、项目背景

AiME 产品于 2026 年 2 月 24 日正式进入 GTM（Go-To-Market）阶段。截至 3 月 18 日，累计签约 237 单，4 名客户成功经理人均服务 59 家客户，远超 1:40 的目标上限。

在此高速增长期，客户成功团队面临三重压力：数据散落在多张 Excel 中难以汇总、周报撰写耗时且质量不稳定、管理层缺乏实时的客户健康度全景视图。每周一份深度周报的生产过程，从数据收集到报告定稿，需要一名资深员工投入整整一个工作日。

本项目的目标是将这一过程产品化——构建一个内部智能体产品，员工只需将原始 Excel 数据丢进指定位置，系统自动完成从数据清洗到深度分析报告的全链路生成。

## 二、商业目标

| 目标维度 | 具体指标 | 衡量方式 |
| --- | --- | --- |
| **效率提升** | 周报生产时间从 1 个工作日降至 30 分钟（含人工审核） | 对比上线前后的实际耗时 |
| **质量标准化** | 每份周报结构一致、指标口径统一、分析深度达标 | 报告结构完整度 ≥95% |
| **决策加速** | 管理层在周四下午即可获得本周全景数据和决策建议 | 报告交付时间稳定在每周四 17:00 前 |
| **知识沉淀** | 历史数据自动积累，支持多周趋势分析和预测 | Board Memory 中连续周数据 ≥4 周后可出趋势图 |
| **可复制性** | 任何员工均可操作，不依赖特定个人的经验和判断 | 新员工 10 分钟内完成首次操作 |

## 四、产品定位

**一句话定位：** 面向客户成功团队的 AI 驱动周报自动化智能体——上传 Excel，自动出报告。

**产品形态：** 基于 YouMind 平台的 Skill + 定时任务 + Board Memory 组合，部署在专属看板中，通过公开链接分发给内部员工使用。

**不做什么：**

- 不做实时数据看板（周报是周频产品，不是日频）

- 不做 CRM 系统（不替代客户管理工具，只做分析和报告）

- 不做外部客户可见的产品（纯内部工具）

## 一、产品概述

### 1.1 产品名称

CSM 周报智能体（内部代号：CSM Weekly Agent）

### 1.2 一句话描述

一个部署在 YouMind 平台上的 AI 智能体，员工将 Excel 数据文件放入指定文件夹后，自动生成包含环比分析、客户健康度分层、异常检测、趋势预测和管理层决策建议的深度客户成功周报，以及配套的交互式数据可视化报告和业务全链路流程图。

### 1.3 产品架构

```plaintext
用户操作层
┌──────────────────────────────────────────────────────┐
│  员工上传 Excel → 指定文件夹                           │
│  （可选）输入本周关键信息摘要文字                        │
│  触发方式：定时自动（周四16:00）或 手动一句话触发         │
└──────────────────────────────────────────────────────┘
                          │
                          ▼
智能体引擎层（Skill，5个Step）
┌──────────────────────────────────────────────────────┐
│  STEP 1: 数据发现 — 自动扫描看板，定位最新数据文件       │
│  STEP 2: 分析引擎 — 清洗、计算、分层、检测、预测         │
│  STEP 3: 报告生成 — 12章节结构化深度周报(Markdown)       │
```

## 三、功能详细规格

### F01: 数据自动发现

**触发条件：** 定时任务到达（每周四 16:00 Asia/Shanghai）或用户手动触发。

**执行逻辑：**

1. 读取 CSM 看板（019d0830-8d82-7c57-b8e9-1fa1c05dc6ed）的 Materials 列表

2. 按 Material Group 定位数据文件：

| Material Group | Group ID | 期望内容 | 匹配规则 |
| --- | --- | --- | --- |
| 03 CS 实施数据 | 019d0915-bb45-7080-a0c0-c025ddee0a19 | 客成实施数据表 | 取该 Group 内最新上传的 Excel |
| 04 用户数据 | 019d0916-3867-72e9-83f3-08e10bbf3ea5 | 用反数据（帖子明细） | 取该 Group 内最新上传的 Excel |
| 根目录或其他 Group | — | 销售数据表 | 文件名含“销售”的最新 Excel |

3. 在 Crafts 列表中定位上期周报：标题含“客户成功周报”的 Page，按创建时间倒序取第一个

4. 读取 Board Memory 中的历史指标快照

**容错规则：**

- 某份 Excel 找不到 → 标注缺失，用已有数据继续执行

- 上期周报找不到 → 标注“首次运行，无环比基准”，跳过环比

- Board Memory 不存在 → 标注“首次运行”，STEP 5 中创建

**输出：** 数据源确认清单（文件名、ID、行数、缺失标注）

---

### F02: 智能表头识别

**问题背景：** Excel 表头字段名可能每周有微小变化（如“客户名称”变成“客户名”，“培训状态”变成“是否已培训”）。

**处理逻辑：**

1. 读取 Excel 第一行作为表头

2. 使用 LLM 语义理解将实际字段名映射到标准字段名

3. 标准字段名定义见本文档第三部分（数据处理逻辑）

**容错规则：**

- 无法识别的字段 → 标注“未识别字段： {原始字段名}”，不影响其他字段处理

- 关键字段缺失（如“客户名称”） → 在报告中标注数据质量警告

---

### F03: 数据清洗

**清洗规则（按优先级执行）：**

| 序号 | 规则 | 说明 |
| --- | --- | --- |
| 1 | 过滤测试数据 | 客户名称字段包含“测试”二字的记录全部剔除 |
| 2 | 去重 | 同一客户在同一表中出现多次时，保留最新一条记录 |
| 3 | 日期标准化 | 所有日期字段统一为 YYYY-MM-DD 格式 |
| 4 | 空值标注 | 关键字段为空时标注，不删除记录 |
| 5 | 异常值标注 | 明显不合理的数值（如负数订单量）标注但不自动修正 |

**输出：** 清洗后的数据集 + 数据质量报告（过滤了多少条、去重了多少条、有多少空值）

### F04: 核心指标计算

**8 个核心指标：**

| 编号 | 指标名称 | 计算公式 | 数据源 | 健康阈值 |
| --- | --- | --- | --- | --- |
| M1 | 累计订单 | COUNT（销售数据全量） | 销售表 | 仅观察趋势 |
| M2 | 本周新增订单 | COUNT（本周签约记录） | 销售表 | 仅观察趋势 |
| M3 | 培训完成率 | 已培训客户数 / 总客户数 | 客成实施表 | 🟢≥80% 🟡60-79% 🔴<60% |
| M4 | 客诉率 | 客诉数 / 已使用客户数 | 客成实施表 | 🟢≤2% 🟡2-5% 🔴>5% |
| M5 | 赔偿率 | 已赔偿数 / 已使用客户数 | 客成实施表 | 🟢0% 🟡≤1% 🔴>1% |
| M6 | 人均持单 | 总客户数 / 客成经理人数 | 客成实施表 | 🟢≤40 🟡40-50 🔴>50 |
| M7 | 产品问题解决率 | 已解决问题 / 总问题数 | 客成实施表 | 🟢≥70% 🟡50-69% 🔴<50% |
| M8 | 客户活跃率 | 有发帖客户数 / 总客户数 | 用反表+客成表 | 🟢≥60% 🟡40-59% 🔴<40% |

**输出格式（每个指标）：**

```plaintext
指标名 | 当前值 | 上周值 | 环比变化(↑↓→) | 变化幅度% | 健康状态(🟢🟡🔴)
```

---

### F05: 环比分析

**数据来源优先级：**

1. Board Memory 中的上周快照（最准确）

2. 上期周报中提取的指标数据（次选）

3. 如果都没有，标注“首次运行，无环比基准”

**计算规则：**

- 环比变化 = （本周值 - 上周值） / 上周值 × 100%

- 方向标注：正变化 ↑、负变化 ↓、无变化 →

- 异常标记：变化幅度 ≥20% 的指标自动进入异常检测队列

---

### F06: 客户健康度分层

**分层标准：**

| 层级 | 色标 | 判定条件（满足任一即归入该层） | 典型特征 |
| --- | --- | --- | --- |
| 健康 | 🟢 | 培训已完成 AND 有发帖记录 AND 无客诉 AND 服务状态正常 | 正常使用，无负面信号 |
| 关注 | 🟡 | 培训未完成 OR 无发帖记录 OR 签约超 2 周未进入使用状态 | 沉默客户，可能流失 |
| 预警 | 🟠 | 有“已安抚”标记 OR 服务状态含“受阻/延迟/待协调” OR 交接信息不完整 | 有潜在风险但尚可控 |
| 危险 | 🔴 | 有客诉记录 OR 有赔偿记录 OR 备注含“不满/投诉/退款” | 已发生问题，需立即干预 |

**判定优先级：** 🔴 > 🟠 > 🟡 > 🟢（客户同时满足多个条件时，取最高风险层级）

**输出：**

- 四层客户数量分布表

- 与上周对比的层级变动明细（哪些客户升级/降级，原因）

- 🔴危险层客户逐一分析卡片（客户名、问题、状态、建议行动）

---

### F07: 异常检测

**触发条件：** 任何核心指标的环比变化幅度 ≥ ±20%

**分析逻辑：**

1. 描述异常现象（“本周客诉率从 4.13% 升至 6.5%，环比+57%”）

2. 下钻数据细节（新增了哪些客诉、涉及哪些客户）

3. 结合用户摘要信息（如有），推测 1-2 个可能原因

4. 给出验证建议（“建议与 XX 客成经理确认是否为集中反馈期”）

---

### F08: 趋势预测

**启用条件：** Board Memory 中有 ≥2 周的历史数据

**预测逻辑：**

- 基于已有数据点的变化方向和加速度，给出下周预测方向（上升/持平/下降）

- 不给出精确数值预测（数据量不足以支撑），只给方向性判断

- 对每个预测标注置信度（高/中/低）

**未启用时的处理：** 在报告中显示“趋势预测将在积累 2 周以上数据后启用，当前已有{n}周数据”

---

### F09: 决策建议生成

**建议分级：**

| 级别 | 定义 | 要求 |
| --- | --- | --- |
| P0 立即行动 | 影响面大且紧迫，不处理会恶化 | 必须具体到“谁、做什么、什么时候、预期效果” |
| P1 本周内 | 需要关注但不紧急 | 明确责任方和行动方向 |
| P2 持续关注 | 中长期风险或优化方向 | 描述风险和建议关注点 |

**生成规则：**

- 每条建议必须有数据支撑（引用具体指标或客户案例）

- 禁止空泛建议（如“建议加强沟通”），必须具体可执行

- 每份报告的 P0 建议不超过 3 条（聚焦最关键的事）

## 六、核心指标计算逻辑（详细版）

### M1: 累计订单

```plaintext
数据源: 销售数据表（清洗后）
公式: COUNT(所有记录)
说明: 统计销售表中的全量记录数，代表自GTM开始以来的累计签约订单
```

### M2: 本周新增订单

```plaintext
数据源: 销售数据表（清洗后）
公式: COUNT(签约日期 ∈ [本周一, 本周日] 的记录)
容错: 如签约日期字段缺失，标注"无法计算本周新增，签约日期字段缺失"
```

### M3: 培训完成率

```plaintext
数据源: 客成实施数据表（清洗后）
分子: COUNT(已培训 = true 的客户)
分母: COUNT(所有客户)
公式: 分子 / 分母 × 100%
输出格式: "51.05% (121/237)"
健康阈值: 🟢≥80%  🟡60-79%  🔴<60%
```

### M4: 客诉率

```plaintext
数据源: 客成实施数据表（清洗后）
分子: COUNT(有客诉 = true 的客户)
分母: COUNT(已培训 = true 的客户)  ← 注意：分母是"已使用客户"，以培训完成作为"已使用"的近似
公式: 分子 / 分母 × 100%
输出格式: "4.13% (5/121)"
健康阈值: 🟢≤2%  🟡2-5%  🔴>5%
特殊说明: 如果未来有"使用客户量"数据，分母应替换为实际使用客户数
```

### M5: 赔偿率

```plaintext
数据源: 客成实施数据表（清洗后）
分子: COUNT(已赔偿 = true 的客户)
分母: 同M4的分母（已使用客户数）
公式: 分子 / 分母 × 100%
输出格式: "1.65% (2/121)"
健康阈值: 🟢0%  🟡≤1%  🔴>1%
```

### M6: 人均持单

```plaintext
数据源: 客成实施数据表（清洗后）
分子: COUNT(所有客户)
分母: COUNT(DISTINCT 客成经理)
公式: 分子 / 分母
输出格式: "59 (237/4人)"
健康阈值: 🟢≤40  🟡40-50  🔴>50
```

### M7: 产品问题解决率

```plaintext
数据源: 客成实施数据表（清洗后）的备注/问题字段 或 独立的问题追踪数据
分子: 已解决问题数
分母: 总问题数
公式: 分子 / 分母 × 100%
输出格式: "41.67% (10/24)"
健康阈值: 🟢≥70%  🟡50-69%  🔴<50%
特殊说明: 问题数据可能需要从备注中提取，或由用户在摘要中补充
```

### M8: 客户活跃率

```plaintext
数据源: 用反数据表 + 客成实施数据表（交叉计算）
分子: COUNT(DISTINCT 用反表中的客户名称)  ← 有发帖记录的客户数
分母: COUNT(客成实施表中的所有客户)  ← 总客户数
公式: 分子 / 分母 × 100%
健康阈值: 🟢≥60%  🟡40-59%  🔴<40%
特殊说明: 需要两张表的客户名称能够匹配（可能存在名称不一致的问题）
```

## 七、环比计算逻辑

### 7.1 上周数据获取优先级

```plaintext
1. Board Memory → 历史指标快照 → 取最近一周的数据
2. 上期周报 → 从"核心指标概览"章节中提取数值
3. 都没有 → 标注"首次运行，无环比基准"，环比列显示"—"
```

### 7.2 环比计算公式

```plaintext
环比变化值 = 本周值 - 上周值
环比变化率 = (本周值 - 上周值) / 上周值 × 100%

方向标注:
  环比变化率 > 0  → ↑ (上升)
  环比变化率 = 0  → → (持平)
  环比变化率 < 0  → ↓ (下降)

特殊处理:
  上周值 = 0 且 本周值 > 0  → 标注"新增"而非计算百分比
  上周值 = 0 且 本周值 = 0  → 标注 → (持平)
```

### 7.3 异常波动判定

```plaintext
IF |环比变化率| ≥ 20%
  → 标记为异常，进入异常检测队列
  → 在报告中用⚠️标注
```

## 八、客户健康度分层计算逻辑

### 8.1 逐客户评估流程

```plaintext
FOR EACH 客户 IN 客成实施数据表:

  # 初始化
  health_level = 🟢健康

  # 检查危险信号（最高优先级）
  IF 有客诉 = true OR 已赔偿 = true:
    health_level = 🔴危险
    GOTO 输出

  IF 备注 包含 ["不满", "投诉", "退款要求"]:
    health_level = 🔴危险
    GOTO 输出

  # 检查预警信号
  IF 服务受阻 = true:
    health_level = 🟠预警
    GOTO 输出

  IF 备注 包含 ["已安抚", "潜在投诉", "待协调"]:
    health_level = 🟠预警
    GOTO 输出

  # 检查关注信号
  IF 已培训 = false:
    health_level = MAX(health_level, 🟡关注)

  IF 客户名称 NOT IN 用反表的客户列表:  ← 无发帖记录
    health_level = MAX(health_level, 🟡关注)

  IF 签约日期存在 AND (今天 - 签约日期) > 14天 AND 已培训 = false:
    health_level = MAX(health_level, 🟡关注)  ← 签约超2周仍未培训

  输出: {客户名称, health_level, 判定原因}
```

### 8.2 层级变动检测

```plaintext
IF Board Memory 中有上周的健康度分布:
  FOR EACH 客户:
    IF 本周层级 ≠ 上周层级:
      记录变动: {客户名, 上周层级, 本周层级, 变动原因}

  变动分类:
    升级（好转）: 🔴→🟠, 🟠→🟡, 🟡→🟢 等
    降级（恶化）: 🟢→🟡, 🟡→🟠, 🟠→🔴 等
```

## 五、STEP 2 技术规格 — 数据清洗、指标计算与深度分析

## 十、错误处理与容错机制

### 10.1 全局错误处理策略

| 错误类型 | 处理策略 | 用户可见行为 |
| --- | --- | --- |
| 数据源缺失（Excel 找不到） | 标注缺失，用已有数据继续 | 报告中注明“XX 数据缺失，以下分析基于已有数据” |
| 数据源为空 | 跳过该数据源 | 报告中注明“XX 数据为空” |
| 上期周报不存在 | 环比列显示“—” | 报告标注“首次运行，无环比基准” |
| Board Memory 不存在 | STEP 5 创建新 Memory | 正常执行，趋势预测标注“数据积累中” |
| Excel 表头无法识别 | LLM 尝试语义匹配，失败则标注 | 报告中列出“未识别字段” |
| 指标计算分母为 0 | 显示“N/A”而非报错 | 报告中标注“分母为零，无法计算” |
| generateWebpage 失败 | 跳过可视化，仅输出周报 | 告知用户“可视化生成失败，周报已完成” |
| edit 回填链接失败 | 在交付清单中单独提供链接 | 告知用户手动添加链接 |
| write 创建文档失败 | 重试一次，仍失败则在对话中输出内容 | 告知用户“文档创建失败，内容已在对话中输出” |

### 10.2 数据质量兜底

当数据质量问题严重到可能影响结论准确性时（如 >30% 的关键字段缺失），智能体应在执行摘要中明确标注数据质量警告，并建议用户核实数据后重新运行。

## 十一、性能与限制

### 11.1 已知限制

| 维度 | 限制 | 应对策略 |
| --- | --- | --- |
| 数据规模 | LLM 上下文窗口限制（约 200K tokens） | 当前客户量级（数百级）可承载；如超过 1000 客户，需引入分批处理 |
| 计算精度 | LLM 数值计算可能有微小误差 | 关键指标建议人工抽检；未来可引入 code-interpreter |
| 实时性 | 依赖 Excel 手动上传 | 未来可对接 CRM API 实现实时数据拉取 |
| 并发 | 单次执行，不支持并行 | 定时任务 + 手动兜底已满足当前需求 |
| 历史深度 | Memory 以文本形式存储，周数增长后体积增大 | 建议保留最近 12 周快照，更早的归档到独立文档 |

### 11.2 未来扩展方向

| 方向 | 说明 | 优先级 |
| --- | --- | --- |
| CRM 数据直连 | 对接 Salesforce / HubSpot API，替代 Excel 上传 | P1 |
| Code Interpreter | 引入代码执行层，提升计算精度和处理大数据集的能力 | P1 |
| 多看板支持 | 支持多个业务线各自独立运行 | P2 |
| 邮件/飞书推送 | 周报生成后自动推送到团队沟通渠道 | P2 |
| 自定义指标 | 允许用户通过配置界面自定义指标公式 | P3 |
| 多语言报告 | 支持中英文双语输出 | P3 |

### STEP 2 指令源码

```plaintext
## STEP 2：数据清洗、指标计算与深度分析

基于 STEP 1 获取的所有数据，执行以下分析。这是整个智能体的核心计算层。

### 2.1 数据清洗
- 过滤客户名称中包含「测试」的所有记录
- 统一日期格式
- 去重（同一客户在同一表中出现多次时，按最新记录为准）
- 标注数据质量问题（如有缺失字段、异常值）

### 2.2 核心指标计算（8 个）

逐一计算以下指标，并从上期周报中提取同口径上周值，计算环比：

| 指标 | 公式 | 数据源 |
|---|---|---|
| 累计订单 | COUNT(销售数据全量) | 销售表 |
| 本周新增订单 | COUNT(本周签约) | 销售表 |
| 培训完成率 | 已培训客户数 / 总客户数 | 客成实施表 |
| 客诉率 | 客诉数 / 已使用客户数 | 客成实施表 |
| 赔偿率 | 已赔偿数 / 已使用客户数 | 客成实施表 |
| 人均持单 | 总客户数 / 客成经理人数 | 客成实施表 |
| 产品问题解决率 | 已解决问题 / 总问题数 | 客成实施表 |
| 客户活跃率 | 有发帖客户数 / 总客户数 | 用反表 + 客成表 |

每个指标输出格式：当前值 | 上周值 | 环比变化（↑↓→）| 变化幅度百分比 | 健康状态（绿/黄/红）

### 2.3 客户健康度分层

将每个客户按以下维度综合评估，分为 4 层：

🟢 健康：培训完成 + 活跃使用 + 无客诉
🟡 关注：培训未完成 或 使用频率偏低 或 签约后沉默
🟠 预警：有潜在投诉 或 服务受阻 或 交接信息断裂
🔴 危险：已客诉 / 已赔偿 / 明确不满

输出：
- 四层客户数量分布
- 与上周对比，哪些客户层级发生了变动（升级/降级），逐一列出并说明原因
- 危险层客户逐一分析（客户名、问题、当前状态、建议行动）

### 2.4 异常检测

扫描所有指标，识别环比变化超过 ±20% 的异常波动。对每个异常：
- 描述异常现象
- 结合数据细节和用户摘要，推测 1-2 个可能原因
- 给出验证建议（如何确认真因）

### 2.5 趋势预测与决策建议

如果 Board Memory 中有 ≥2 周的历史数据，基于趋势给出：
- 下周关键指标预测方向
- 管理层决策建议，按优先级分类：
  - P0（立即行动）：影响面大且紧迫
  - P1（本周内）：需要关注但不紧急
  - P2（持续关注）：中长期风险

如果历史数据不足 2 周，标注「趋势预测将在积累 2 周以上数据后启用」。

### 2.6 未闭环事项追踪

从上期周报中提取所有「需要决策的事项」「风险预警」「下周重点」，逐一检查本期数据中是否已闭环：
- ✅ 已闭环：标注解决方式
- 🔄 进行中：标注当前进展
- ⬆️ 升级：问题恶化，需要更高层级关注
- ❓ 无法判断：数据中未体现，需人工确认

所有分析结果暂存，供 STEP 3 撰写报告使用。
```
